A replica of the dashboard at localhost:3838 — running on a script.

01 — pace

How fast, not how full.

76% used means nothing on its own. 76% with only 69% of the week elapsed means you run dry before the reset. Burndown does that second step and puts the verdict in the menu bar: a multiple of the sustainable rate, and where you'll land at reset — green when it fits, 🔥 when it won't (and when you'll hit the wall).

🟢 proj 78% sustainable · right on your Tuesday 🟡 proj 92%↑ tight — heavier than usual 🔥 proj 140% · dry in 31 h
02 — projection

It knows you don't burn quota while you sleep.

A naïve line holds your current pace flat — so a busy afternoon reads as a runaway week. The smart projection walks each hour left to the reset and applies your own learned burn for that weekday and hour, modelling quiet nights and weekends from your history. Until a cell has data, it falls back to recent pace and says which it's using.

Weekly limit — 30% used naive (flat pace) → proj 465% dry tomorrow smart (learned) → proj 92% fits · 8% headroom
03 — surge alarm

A runaway bill, caught while you sleep.

OpenRouter credit is real money, so a buggy process can burn it fast. When spend over the last half hour jumps far above your normal and would drain the balance quickly, a surge alert pushes to your phone over ntfy — and because a leak at 3am is exactly when you can't see it, this one overrides quiet hours. Brief bursts are averaged out, so it only trips on a real surge.

🚨 OpenRouter spend surge 03:14 $48.20/h, ~70× your normal — balance empties in ~14 min. Runaway process?
04 — token history

Real tokens, from your own transcripts.

The usage API only gives opaque %-of-limit windows. The dashboard reads your local Claude Code transcripts for the actual per-message token counts, so you get true usage by model, project, and day — deduped and scanned incrementally. Nothing is sent anywhere; it's already on disk.

this week · input + output +107% vs last opus-4-8 ████████████████ 35.9M opus-4-7 ██████████▏ 24.1M fable-5 ██ 4.1M sonnet ▏ 0.2M
05 — memory

Every shift, remembered.

It learns what a typical Tuesday 3pm looks like and flags when now is out of line. It logs each episode — a spend surge with its peak and dollars burned, a weekly exhaustion, an early reset Anthropic slipped in — as an incident with a start and an end. And it leaves those abnormal periods out of the baseline, so a holiday spike never becomes your new normal.

00:14 🚨 surge peak $48/h · 22 m · -$3.10 Sat ⛔ weekly exhausted — waited on reset Sat ↺ reset early (Anthropic)
how it works

Reads what's already there. Nothing leaves.

usage API ───┐ ~/.claude ───┼──→ analytics ──┬──→ menu bar keychain ───┘ (one file) ├──→ dashboard :3838 ──→ charts └──→ ntfy ──────────────→ phone

The plugin polls the same endpoint Claude Code's /usage uses, reconstructs each window's burn rate from snapshot deltas, and learns your rhythm — all in one stdlib Python file. The dashboard reuses that exact math; it only ever reads. Two HTTPS requests leave the machine — api.anthropic.com and openrouter.ai — plus the ntfy push you opt into.

install

One file for the bar. One command for the dashboard.

# macOS · SwiftBar · Python 3 (no dependencies)
brew install swiftbar
curl -fsSL https://raw.githubusercontent.com/alcides-collective/burndown-bar/main/burndown-bar.5m.py \
  -o "$(defaults read com.ameba.SwiftBar PluginDirectory)/burndown-bar.5m.py"

# optional: web dashboard + phone alerts (starts on login)
cd dashboard && ./install.sh        # → localhost:3838

macOS only. The menu bar needs SwiftBar and a Claude Pro/Max subscription logged into Claude Code. The dashboard adds a small React + Motion + uPlot front-end over a stdlib Python backend. System fonts throughout — no web fonts, no trackers.
